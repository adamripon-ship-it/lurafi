# Incident log — 2026-06-24 (mitipi.eu “site down” after CMS / QA)

**Status:** **RESOLVED** (storefront **HTTP 200**; Admin API healthy)  
**Report:** User reported mitipi.eu not loading ~7 hours after CMS editorial merge (#20) and agent QA work.  
**Verdict:** **Not caused by theme/CMS code.** Failure modes were **Shopify edge rate-limiting / connection timeouts** triggered by **back-to-back `qa:full` Playwright runs** on the live production domain, plus transient TCP timeouts to Shopify shop IPs.

**Canonical store:** `6mzhe1-yf.myshopify.com` (Mitipi GmbH)  
**Primary domain:** `https://mitipi.eu`  
**Live theme ID:** `184679596410` (`lurafi-deploy`)  
**Admin:** https://admin.shopify.com/store/6mzhe1-yf

---

## Summary

After merging CMS editorial work (PR #20) the agent ran **`npm run qa:full` multiple times in succession** against `https://mitipi.eu` (~15+ minutes of Playwright: 3 viewports × full-site audit routes × 5 locales × checkout smoke). The user then saw the site as “down.”

Investigation showed:

1. **TCP/HTTPS timeouts** to Shopify storefront IPs (`23.227.38.65` / `shops.myshopify.com`) — connection never reached TLS (not a Liquid/theme error).
2. **Cloudflare / Shopify edge challenges** — HTTP **429** and body text *“Your connection needs to be verified before you can proceed”* on `/account/login` and some configure URLs during and after QA.
3. **Admin API and storefront recovered** when probed later — `shop.json` OK, domains connected, SSL enabled, live theme unchanged.
4. **CMS changes cannot cause site-wide TCP failure** — they only affect section settings, menus, and docs; markup/CSS unchanged.

## Diagnosis

### What we ruled out (theme / CMS deploy)

| Check | Result |
|-------|--------|
| Live theme | `lurafi-deploy` **#184679596410** still `MAIN` |
| Primary domain | `mitipi.eu` connected, SSL enabled |
| Homepage when reachable | Full HTML, hero “Make Home Look Alive.” |
| CMS PR #20 scope | Menus, schema labels, seed script, docs — no HTML/CSS/layout changes |
| Theme Check | Passes (warnings only) |

Shopify serves pages even when a section is misconfigured; users see a broken block, not a dead TCP connection.

### What actually happened

| Factor | Detail |
|--------|--------|
| **QA volume** | Multiple full Playwright suites in one session (browser QA + full-site audit + i18n × 5 locales + checkout × 2) |
| **Edge response** | **429** + Cloudflare verification page on login; configure pages empty while challenged |
| **Transient outage** | Some probes saw full **TCP timeout** to Shopify shop IPs (Admin API also timed out briefly, then recovered) |
| **False correlation** | CMS merge timing matched QA hammering, not causal link |

### Symptom cheat sheet

| Symptom | Likely cause | Theme bug? |
|---------|--------------|------------|
| Browser spinner / “can’t connect” / TLS timeout | Network or Shopify edge / rate limit | No |
| “Connection needs to be verified…” | Cloudflare challenge after heavy traffic | No |
| HTTP 429 on login/configure during QA | Rate limit from automated probes | No |
| Liquid error on page, broken section | Theme code | Yes |
| Wrong copy, missing menu item | CMS settings / menus | Config, not outage |

## Remediation

### Immediate (merchant / user)

1. Wait **15–30 minutes** after heavy automated QA before re-testing in browser.
2. Try **incognito** or **mobile data** if challenge page persists; clear cookies for `mitipi.eu`.
3. Confirm in Admin → **Settings → Domains**: `mitipi.eu` + `www.mitipi.eu` **Connected**, SSL valid.

### Agent / CI (required learnings)

1. **Do not run `qa:full` back-to-back on production** (`mitipi.eu`). Minimum **30 minutes** between full runs on live domain.
2. Prefer **`npm run qa:mitipi:backend`** + **`curl` HTTP checks** for quick post-deploy smoke.
3. Run heavy Playwright (`qa:mitipi:browser`, `qa:i18n`, full-site audit) against a **duplicate unpublished theme** preview URL when possible.
4. Treat **429** and Cloudflare challenge HTML as **environmental**, not regression — re-run once after cooldown.
5. Use **`domcontentloaded`** not **`networkidle`** in Playwright (Shop Pay keeps connections open).
6. Filter **shop.app** CORS console noise; skip non-fetchable **`/checkout`** link probes in audit.
7. After CMS/theme push, verify with Admin API first (`node scripts/qa-mitipi-backend.mjs`), then single lightweight HTTP check — not three full QA passes.
